5. Compare the simulated versions with the original version of your game

One of the advantages of using color blindness simulations to identify problems is the possibility of comparing simulated versions of the interface with the original version, without simulation. This can be a good starting point to look for color accessibility problems, guided by the questions: are the elements I see in the original version also visible in the simulated versions? Do elements that are different in the original version look the same in the simulated version?

PROBLEM SCENARIO

You use color blindness simulations to look for color accessibility problems in your games, however, you have difficulty identifying problems even with the help of guidelines.

Our study demonstrates that participants using simulations were able to identify a greater variety of color accessibility problems than participants with color blindness. This may have happened because the simulations show the approximate effects of more than one type of color blindness, while participants with color blindness could only rely on their own vision to identify problems.

USE SCENARIO

You are developing a game interface and decide to check for color accessibility problems using a color blindness simulation. You use a simulation on your desktop screen to identify problems and make adjustments in real time. You switch between the original unfiltered view and each of the simulated views, initially looking at the interface in its entirety. Then, you do the same but now observing each element of the interface individually (e.g.: buttons, characters, obstacles). You find some elements that are heavily affected by the simulations and make adjustments to the colors used to improve contrast.

Remember that the earlier in the process color accessibility problems are identified and resolved, the lower their cost will be in terms of time and project resources.

TIPS

  • Initially look for visualization problems in the interface in general by analyzing the relationships between interface elements. E.g.: “This button, in the protanopia simulation, is exactly the same color as the main character's clothes. Could this cause any confusion?”
  • It is also important to analyze the elements individually, checking the contrast with the different backgrounds that may appear during the game. E.g.: “The main character’s clothing blends into one of the backgrounds when viewed with the deuteranopia simulation.”
  • Take advantage of comparing the simulated versions with the original version of the game to try to identify parts of the interface that are invisible to players with color blindness. If identified, these areas should be explored in tests with players with color blindness (more information in recommendation 8)
  • Remember, the interface of a mobile game is not usually static and will not be used on the desktop. Whenever possible, try to test the game in its playable version, to also take into account animations and transitions, and on the target device.
  • If you are unsure whether there is sufficient contrast in the chosen colors, you can use contrast analysis tools in both the original and simulated versions of your game.
  • Take note of any pain points identified through the simulations to consult players with color blindness. Even if you have modified the colors to improve visualization or differentiation, it is important to confirm with the players that the changes made were sufficient. More information in Recommendation 7.

WHAT NOT TO DO

  • Don't just analyze graphical elements in isolation. Many chromatic accessibility problems are related to the differentiation of elements during gameplay.
